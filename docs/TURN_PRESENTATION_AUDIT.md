## 타임라인 계측

작업 전 소스의 기본 모드는 주사위 약 1,000ms, 이동 칸당 120~160ms(최대
1,500ms), 도착 후 별도 확인 0ms, 단일 경제 변화 약 520ms, 결과 요약 0ms였다.
서버 응답과 폴링이 먼저 도착하면 각 사건의 원인이나 다음 행동을 읽기 전에 버튼이
갱신되는 것이 핵심 원인이었다. 작업 전에는 구간별 계측 자체가 없어 이 값은 당시 코드의
실행 경로와 타이머를 재구성한 기준값이며, 아래 값부터 개발 모드의 브라우저 실측값이다.

Chromium에서 무주 일반지역 도착을 기본 모드로 1회 계측한 결과:

| 구간 | 측정값 |
|---|---:|
| 서버 응답 | 13ms |
| 주사위 공개 | 1,295ms |
| 말 이동 | 388ms |
| 도착 공개·유지 | 903ms |
| 경제 결과 | 0ms |
| 결과 요약 | 608ms |
| 클릭 후 입력 활성화 | 3,276ms |

개발 모드에서는 `window.turnPerformanceLog` 최근 30건과 JSON `console.info`에
`action_id`, 서버 응답, 주사위, 이동, 도착, 경제, 입력 활성화, `scene_order`,
`state_version`을 기록한다. 운영 화면에는 로그를 출력하지 않는다.

## 장면 상태와 기본 시간

표현 상태는 서버 상태와 별도로 `ACTION_REQUEST → DICE_REVEAL → PIECE_MOVEMENT →
ARRIVAL_REVEAL → ECONOMIC_RESULT/EVENT_REVEAL → RESULT_SUMMARY →
PLAYER_DECISION/TURN_COMPLETE` 순서로 관리한다. Promise 큐가 순서를 보장하며 장면 중
폴링 결과는 `pendingSnapshot`에 보관한다. 게임 종료, 초기화, 인증 상실, 자동 퇴장,
파산, 새 게임 인스턴스는 즉시 반영한다.

| 장면 | 기본 정책 |
|---|---|
| 주사위 | 굴림·결과 합계 최소 1,200ms, 최종 면 350ms 이상 |
| 이동 | 짧은 경로 170ms/칸, 긴 경로 140ms/칸, 전체 1,600ms 이하 |
| 도착 | 일반 700ms, 선택 발생 900ms, 출발지 1,200ms, 이벤트 전 500ms |
| 경제 | 단일 750ms, 이체 1,000ms, 복수 350ms/항목, 5개 이상 1,500ms |
| 출발지 정산 | 4,000ms 이상, 항목별 공개, 건너뛰기 가능 |
| 결과 요약 | 600ms 이상; 출발지와 중대한 결과는 명시 확인 |

속도 배율은 여유롭게 130%, 기본 100%, 빠르게 60%, 모션 최소화 0%다. 변경은 다음
장면부터 적용된다. `fast_simulation`에서는 모든 표현 배율을 0으로 두며 서버에는
연출용 `sleep`을 추가하지 않았다.

## 입력과 결과 전달

상태 변경 버튼은 서버 응답, 주사위, 이동, 도착, 필수 경제 결과, 이벤트 확인, 최신
`state_version` 동기화가 모두 끝나고 `PLAYER_DECISION`에 진입할 때만 활성화된다.
정보 탭과 도움말은 계속 사용할 수 있다. 잠긴 버튼에는 이동 중, 경제 결과 표시 중,
이벤트 확인, 최신 상태 동기화 등 실제 이유를 표시한다. 서버도 필수 `pending_action`이
있으면 턴 종료를 거부하고 `allowed_actions.end_turn`을 비활성화한다.

결과 카드는 도착/행동 결과, 금액, 현재 현금, 서버가 정한 다음 행동을 순서대로 보여
준다. 경제 화면은 원인과 대상 → 송금 방향 → 금액 → 최종 현금 순서다. 출발지·파산·
긴급대출·특수지역 강제매각은 명시 확인 대상으로, 작은 단일 변화는 최소 시간이 지나면
자동 진행 대상으로 분류한다. 사용자는 언제든 계속 또는 연출 건너뛰기를 선택할 수 있다.

## 도착 유형별 흐름

- 무주 일반지역: 지역·무소유 상태 → 구매 안내 → 구매/포기.
- 타인 일반지역: 소유자·건물 → 비용 원인과 이체 → 현금 → 턴 종료 안내.
- 내 일반지역: 보유 건물·상태 → 서버가 허용한 건설·관리·거래·턴 종료.
- 특수지역: 현재 가치 → 가치 상승/강제매각 → 금액 → 후속 행동.
- 출발지: 수익 → 손실 → 과세소득 → 세금 → 보너스 → 대출 상환 → 신규 대출/파산 → 최종 현금.
- 이벤트: 도착 강조 → 카드 공개 → 핵심 효과 → 경제 변화 → 확인 → 다음 행동.

출발지 서버 응답의 `settlement_steps`는 `building_income`, `building_loss`,
`taxable_income`, `tax`, `start_bonus`, `loan_repayment`, `new_loan`, `final_cash`
순서를 보장한다. 정산 전 현금, 총수익, 총지출, 정산 후 현금도 함께 반환한다.

## 봇과 이벤트 타이머

`bot_action_delay`는 서버에서 봇 턴 시작 전 판단 대기에만 적용된다. 서버의 한 턴 판정은
원자적으로 유지한다. 클라이언트 표현 정책은 `turn_start_ms=400`, `decision_ms=600`,
`result_hold_ms=500`, `between_turns_ms=600`이며 서버 지연으로 사용하지 않는다. 표현 큐는
10개로 제한하고 오래된 단순 주사위 장면만 압축한다. 경제·구매·건설·파산·이벤트는
버리지 않는다. 빠른 시뮬레이션은 모두 즉시 처리한다.

이벤트 카드 뒤집기와 확인 네트워크 요청 동안 서버 턴 시간을 제외한다. 카드 공개 후
읽기 시간은 정상 경과하며 남은 시간이 10초 이하이면 즉시 공개한다. 미확인 이벤트가
있는 제한시간 종료는 무행동 누적/자동 퇴장으로 계산하지 않는다. 이 정책은 공식 규칙
`EVENT-011`과 구현 매트릭스에 추가했다.

정적 계약은 9개 장면, 네 속도 배율, 최소 시간, 결과 카드, 성능 필드를 검사한다. 엔진
테스트는 필수 결정 중 턴 종료 거부, 이벤트 연출 시간 제외와 읽기 시간 경과, 이벤트 확인
중 자동 퇴장 방지, 출발지 정산 순서를 검사한다. Chromium 통합 테스트는 호스트 1개와
플레이어 4개 독립 컨텍스트에서 무주 이동 → 구매 → 구매 후 건설 → 매각 → 재접속을
수행하고 최초 장면 순서, 2~7초 입력 잠금, 서버 500과 콘솔 오류 0건을 검증한다.

실제 Android Chrome, Samsung Internet, iOS Safari에서는 저사양 기기의 프레임 드롭,
백그라운드 복귀, 화면 읽기 속도를 확인해 칸당 이동과 경제 항목 유지 시간을 조정해야 한다.
